## 一、开发目标（总体）

### 1.1 项目定位

本项目是**企业级 AI Work Control Center（AI 工作中控后台）**，目标是构建可扩展的 AI 控制平面，用于管理：

- 租户
- 项目
- Agent
- Skills
- Workflow
- Terminals
- Tasks

### 1.2 当前阶段：第一阶段「控制平面骨架」已完成

重点在于：

- 双后台结构（平台后台 + 租户后台 + 系统管理员区）
- 多租户基础
- 权限体系
- UI 结构
- 模块边界
- 标准接口
- 服务端持久化（Prisma + SQLite）
- 服务端 LLM 执行链路

**第一阶段明确不做的内容：**

- 接入真实 Facebook / X / TikTok
- 接入 OpenClaw / 复杂 Agent Runtime
- 实现真实任务自动执行
- 复杂拖拽式流程设计器
- 在页面组件中编写复杂业务逻辑

### 1.3 架构原则

- **控制平面优先**：先完成管理结构、UI 结构、模块边界、接口规范，再接执行逻辑
- **mock-first**：前端 mock 数据与服务端 API 并存，逐步迁移
- **分层结构**：page → service → repository → schema，页面不直接依赖 mock
- **可扩展性**：为 Agent runtime、Workflow engine、Skills execution、Terminal connectors 预留扩展位

### 2.8 LLM 执行链路（已完成）

```
前端 llmExecutor
  → POST /api/llm/execute
  → llmResolverService（AgentLLMBinding → LLMModelConfig → LLMProvider → LLMCredential）
  → openaiProviderServer（读取凭证密钥，真实调用 OpenAI）
```

**配置方式**：以平台管理员身份登录 → `/system/llm-configs` → 创建凭证（填写 OpenAI API Key）→ 绑定到 Provider → Provider 绑定到 ModelConfig → ModelConfig 绑定到 AgentTemplate。

## 三、已解决的问题

### 3.1 模型配置页面空白（Phase 17.7a）

**原因**：`llmCredentialService` 改为异步 API 后，页面 `load()` 仍同步调用，将 Promise 当作数组传给 Table。  
**修复**：`LLMConfigCenterPage` 的 `load` 改为 async，所有操作改为 await。

### 3.2 凭证存储方式

**需求**：密钥不能写入 `.env`，需通过后台录入并加密存储。  
**实现**：新增 `server/data/credentialStore.ts`，凭证落盘到 `credentials.json`（加入 `.gitignore`），前端只展示掩码，完整密钥仅在服务端读取。

### 3.3 Provider / ModelConfig / Binding 前后端数据分离（Phase 17.7b）

**原问题**：前端模型配置中心的增删改只影响内存 mock，服务端 `serverStore.ts` 为硬编码数组，二者完全独立。  
**修复**：
- 新增 Prisma 模型（AgentTemplate / Skill / LLMProvider / LLMModelConfig / AgentLLMBinding）
- 新增 `server/domain/llmConfigDb.ts` 完整 CRUD
- 新增 `/api/llm-providers`、`/api/llm-model-configs`、`/api/agent-llm-bindings` REST API
- 前端 3 个 tenant repository 改为调用上述 API
- `llmProviderService`、`llmModelConfigService`、`llmBindingService` 全部改为 async
- 前端模型配置中心所有增删改均实时持久化到服务端

### 3.4 AgentTemplate / Skill 同步到服务端（Phase 17.7b）

- 新增 Prisma AgentTemplate / Skill 模型，提供完整 CRUD API
- 前端 `agentTemplateRepository`、`skillRepository` 改为调用 `/api/agent-templates`、`/api/skills`
- 服务启动时自动播种：12 条 AgentTemplate 种子 + 16 条 Skill 种子 + LLM 配置种子

### 3.6 终端中心完整化（Phase 17.8）

**终端模块从占位升级为完整治理系统：**

| 层 | 原状态 | Phase 17.8 完成状态 |
|---|---|---|
| System Shell | 无终端管理入口 | ✅ 终端类型工厂（列表/新建/详情/编辑），11 条预置种子数据 |

### 终端类型工厂 UX 流程

```
System Admin 进入 /system/terminal-types
→ 查看 11 条预置终端类型（api/browser/mcp 三类）
→ 可新建自定义终端类型（填写 authSchema / configSchema JSON Schema）
→ 可停用/启用（影响租户侧向导中该类型的可选性）
→ 可删除（有租户实例时拒绝）
→ 详情页查看认证/配置字段定义 + 能力标签 + 适用项目类型 + 租户使用数
```

### 租户新建终端向导 UX 流程

```
Tenant User → 终端中心 → 新建终端
→ Step 1：从 active SystemTerminalType 中选择类型（按 api/browser/mcp 分组）
→ Step 2：填写终端名称 + 动态认证字段（authSchema → parseSchemaFields）+ 动态配置字段（configSchema）
→ Step 3：点击「测试连接」验证凭证（当前 mock 成功）
→ Step 4：可选绑定身份 + 多选关联项目
→ 确认创建 → 跳回终端列表
```
