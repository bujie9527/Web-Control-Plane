# AI Work Control Center - 系统总览

## 1. 文档目的

本文档用于定义 AI Work Control Center 第一阶段的系统总体定位、核心目标和整体结构。

它是本项目的高层架构参考文档，供以下场景使用：

- 产品规划
- 架构设计
- Cursor 开发约束
- 新模块接入参考
- 团队统一理解

---

## 2. 产品定位

AI Work Control Center 是一个企业级 AI 工作中控后台。

它不是聊天工具，不是单个 Agent 演示系统，也不是单纯的自动化脚本面板。

它的定位是：

**一个管理 AI 团队、业务项目、流程能力、执行终端与结果反馈的控制平面。**

---

## 3. 第一阶段目标

第一阶段只做“控制平面骨架”，重点包括：

1. 双后台结构  
   - 平台后台（Platform Console）
   - 租户后台（Tenant Console）

2. 多租户基础能力  
   - 租户管理
   - 租户成员管理
   - 多租户隔离

3. 后台管理基础结构  
   - 菜单系统
   - 路由结构
   - 页面骨架
   - 统一 UI 风格

4. 标准化工程基础  
   - 模块目录规范
   - mock service
   - schema / service / repository 分层
   - API 返回结构
   - event / audit 预留接口

---

## 4. 当前阶段不做的内容

为了保证项目架构稳定，第一阶段不做：

- 真实 Facebook / X / TikTok / WhatsApp / CMS 接入
- 真实 OpenClaw / Agent Runtime 接入
- 真实 Workflow Engine 执行
- 复杂拖拽式流程设计器
- 复杂数据库与分布式架构实现
- 高复杂度业务自动化逻辑

当前重点是：

**先把“后台控制平面”搭建好。**

---

## 5. 系统总体结构

系统分为四层：

### 5.1 管理层
负责系统组织、角色、权限、租户、菜单、系统设置。

### 5.2 业务层
负责项目、目标、渠道、流程、任务、结果等业务对象。

### 5.3 能力层
负责 Agent、Skills、模板、终端能力、后续 Workflow 执行能力。

### 5.4 执行层
未来用于承接终端操作、任务运行、第三方平台连接、自动化执行。

在第一阶段，执行层只预留结构和接口，不做真实接入。

---

## 6. 双后台结构

### 6.1 平台后台（Platform Console）

平台后台用于平台运营方管理整个平台。

核心职责：

- 平台工作台
- 租户管理
- 平台用户管理
- 资源与配额
- 模板中心
- 审计日志
- 平台设置

平台后台不负责租户业务执行。

---

### 6.2 租户后台（Tenant Console）

租户后台用于企业租户管理自身的业务工作。

核心职责：

- 工作台
- 项目中心
- 任务执行
- 流程中心
- Agent 中心
- Skills 能力库
- 终端中心
- 数据分析
- 系统管理

租户后台是本项目后续商业化和业务演示的核心。

## 8. 架构原则

### 8.1 控制平面优先
先做管理结构、页面骨架、权限与接口规范，再接业务执行能力。

### 8.2 双后台彻底分离
平台后台与租户后台必须从路由、菜单、布局、模块边界上分离，不能靠角色隐藏菜单来模拟。

### 8.3 mock first
第一阶段使用 mock data / mock service 构建真实页面结构和交互感受。

### 8.4 分层清晰
页面、schema、service、repository、provider 分层清晰，不把逻辑堆进页面组件。

### 8.5 组件复用
优先复用统一布局、表格、卡片、状态标签、抽屉等通用组件。

### 8.6 预留扩展能力
后续可接入：
- Agent runtime
- Workflow engine
- Terminal connectors
- 第三方平台能力

---

## 9. 本阶段的成功标准

第一阶段完成后，系统应达到以下标准：

1. 看起来像完整的企业级管理后台
2. 平台后台与租户后台结构清晰
3. 菜单层级和业务路径合理
4. 页面风格统一
5. 项目详情页具备“工作台感”
6. 后续接入真实能力时无需推翻现有架构
