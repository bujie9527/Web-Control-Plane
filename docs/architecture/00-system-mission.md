# AI Work Control Center — 系统使命与底层目标

## 一、系统定位

**AI Work Control Center** 是一个**以项目为驱动、以垂直能力 Agent 为执行单元、以身份（Identity）定义内容视角、以终端为操作入口的 AI 运营控制平面。**

它解决的核心问题是：**让企业可以用 AI 团队，持续运营多个平台账号和网站，以不同的身份视角产出差异化内容，并产出可向上汇报的业务结果。**

系统不是工具，不是聊天机器人，而是一个**可运营的 AI 组织**。

---

## 二、核心项目类型

系统当前支持以下项目类型，每种类型有独立的目标模式、交付标准和推荐流程：

| 项目类型 | 业务目标 | 典型交付物 | 主要终端类型 |
|---|---|---|---|
| **自媒体内容运营** | 持续在平台发布内容，积累粉丝与曝光 | 发布内容数、互动数据、涨粉数 | 微博、小红书、TikTok、Instagram、X 等 |
| **账号矩阵管理** | 同一平台多账号差异化运营，形成规模效应 | 矩阵账号发布率、内容差异度、整体曝光 | 同平台多账号（每账号设置独立身份） |
| **网站运营与 SEO** | 维护网站内容更新，提升搜索排名与自然流量 | 发布文章数、关键词排名、页面收录量 | CMS / WordPress / 网站 API |
| **线索获客** | 引导目标用户主动联系，进入私域沉淀 | 新增私域联系人数（WhatsApp / Line 加粉量） | WhatsApp Business、Line 官方账号、落地页 |

**扩展原则：** 新业务类型只需新增项目类型配置包（目标类型 + 流程模板 + 推荐 Agent 组合），不改底层架构。

## 三、核心概念定义（必须准确理解，严禁混用）

### 3.1 身份 / Identity

**Identity 属于项目层，定义内容创作的视角与立场。**

Identity 不是 Agent 的属性，而是项目中"内容以谁的口吻产出"的设定。

**核心比喻：** 美联储发布一项经济政策，由「日本经济学家身份」产出的评论，与由「美国经济学家身份」产出的评论，内容、立场、语气完全不同——这就是 Identity 带来的差异化。

一个项目可以配置多个 Identity，对应多个账号或网站，每个 Identity 对应一组终端。Identity 是租户的长期资产，可跨项目复用。

**Identity 包含：**
- **核心定位**：我是谁、我的角色与背景
- **语气与风格**：表达方式，正式/活泼/专业/亲切等
- **内容方向**：聚焦哪些主题，擅长什么领域
- **行为规则**：不说什么，如何回应争议，边界在哪里
- **平台适配**：同一身份在不同平台的差异化表达策略

---

### 3.2 Agent（智能体）

**Agent 是具备垂直能力的 AI 执行单元，按领域或平台划分，不承载人格。**

Agent 的能力来自 **LLM + Skills 的组合**。不同 Agent 对应不同的工作领域，多个 Agent 在流程中协同，各自负责流程节点中自己擅长的部分。

**Agent 能力来源：**
```
Agent = LLM（理解与生成能力）+ Skills（领域操作最短路径）
```

**典型 Agent 类型示例：**

| Agent | 能力边界 |
|---|---|
| Facebook 内容创作 Agent | 生成适合 Facebook 平台的内容格式与风格 |
| 小红书内容创作 Agent | 生成小红书图文 / 种草内容 |
| SEO 文章创作 Agent | 关键词布局、结构化长文写作 |
| 网站操作 Agent | 登录 CMS、发布文章、更新页面 |
| 服务器运维 Agent | 执行服务器配置与部署流程 |
| 线索引导 Agent | 生成私域引流内容与落地页文案 |

**重要约束：**
- Agent 执行任务时，接收来自项目的 Identity 信息作为内容视角输入
- Agent 本身不定义"我是谁"，Identity 才定义这一点
- Agent 只决定"我有什么能力"，Identity 决定"内容从什么角度产出"

---

### 3.3 Skills（技能）

**Skills 是完成特定任务的最短路径，是 Agent 的行动指令库。**

Skills 解决的问题是：不让 AI 每次都"摸索"如何完成一件事，而是给它一份经过验证的操作流程，直接执行。

**Skills 的本质：** 人类经验的结构化沉淀，描述"做这件事最高效的步骤是什么"。

**典型 Skills 示例：**
- 「Facebook 发帖流程」→ 进入主页 → 点击创建帖子 → 填写内容 → 选择受众 → 发布 → 确认状态
- 「小红书 SEO 标题优化」→ 提取核心关键词 → 套用爆款标题公式 → 校验字符数限制
- 「WordPress 文章发布」→ 登录后台 → 新建文章 → 填写标题/正文/分类/标签 → 发布
- 「WhatsApp 私域引流」→ 生成行动号召文案 → 附上联系方式 → 发布到指定渠道

**OpenClaw 兼容：** Skills 的格式与 OpenClaw 等 AI 操作系统的 Skill 格式对齐。可以直接从 OpenClaw 生态复制已有 Skills，快速扩充 Agent 能力库，不需要从零重新定义。

---

### 3.4 终端 / Terminal

**Terminal 是 Agent 操作真实世界的入口，系统的能力边界由已接入的终端决定。**

每个终端绑定到一个 Identity 下，代表该身份在某个平台的实际账号或操作入口。

**终端类型示例：**
- 社交媒体账号（Facebook Page、Instagram、TikTok、小红书）
- 通讯账号（WhatsApp Business、Line 官方账号）
- 网站后台（WordPress、自定义 CMS API）
- 落地页系统

**约束：** Agent 只能操作已配置并绑定到项目 Identity 下的终端，不得操作未授权终端。

## 四、核心执行链路

```
项目（Project）
  ├── 设定：目标（Goal）+ 交付标准（Deliverable）
  ├── 设定：身份（Identity）× N  ← 内容以谁的视角产出
  │         └── 绑定终端（Terminal）← 这个身份操作哪些账号/网站
  │
  ├── SOP（人写的操作规范，自然语言）
  │     ↓ Planner Agent 解析
  ├── 流程模板（WorkflowTemplate）
  │     └── 节点（WorkflowNode）× N
  │           └── 绑定 Agent（按能力匹配）
  │                 └── Agent = LLM + Skills
  │
  └── 执行结果（Result）→ 汇报视图（Report）
```

**节点执行时的内容生成逻辑：**
```
WorkflowNode 触发
  → Agent 接收任务（任务指令 + 项目 Identity + 终端信息）
  → Agent 调用 Skills（获取操作最短路径）
  → Agent 通过 LLM 生成内容（以 Identity 的视角和风格）
  → Agent 通过 Terminal 执行操作（发布 / 更新 / 提交）
  → 结果回写到项目
```

## 五、七条核心设计原则

以下原则为系统架构红线，所有开发必须遵守，不得违背：

### 原则 1：项目是一切执行的起点
所有工作流、Agent 分配、身份配置、终端操作，都必须归属于一个项目。禁止游离于项目之外的孤立执行。

### 原则 2：项目类型决定配置包
项目类型绑定可用的目标类型、推荐流程模板、推荐 Agent 组合。新增业务场景以新增项目类型的方式扩展，不修改已有类型逻辑。

### 原则 3：Identity 是内容差异化的来源，属于项目层配置
Identity 决定"内容以谁的立场产出"，与 Agent 的能力无关，不是 Agent 的属性。同一个内容创作 Agent，在不同 Identity 设定下产出完全不同的内容。Identity 是租户的长期资产，可跨项目复用。

### 原则 4：Agent 是垂直能力单元，能力来自 LLM + Skills
Agent 按领域/平台划分，不承载人格。Agent 的核心竞争力是 Skills 库的质量。Skills 描述完成任务的最短路径，与 OpenClaw 格式兼容，可快速从外部生态复用。

### 原则 5：终端是执行的边界，系统能力由终端决定
Agent 只能操作已配置并绑定到项目 Identity 下的终端。系统能做什么，取决于接入了哪些终端，不硬编码业务能力。

### 原则 6：SOP → Workflow 转换是核心 AI 价值点
SOP 是人写的操作规范（自然语言），Workflow Template 是 Planner Agent 将其解析为可执行节点结构的产物。这条转换链不可绕过，是系统将"人的经验"转化为"可复用流程"的关键机制。

### 原则 7：结果必须可向上汇报
项目执行结果需产出汇报视图：完成了哪些交付物、达成了哪些目标指标、各身份/账号的产出数据。这是系统对外的核心交付价值。

## 六、关键领域对象关系

```
Tenant（租户）
  └── Project（项目）[类型驱动配置]
        ├── Goal（目标）+ Deliverable（交付标准）
        ├── Identity（身份）× N  [内容视角，项目层资产]
        │     └── Terminal（终端）× N  [绑定平台账号/网站]
        ├── SOP → WorkflowTemplate → WorkflowInstance
        │         └── WorkflowNode → Agent（按能力匹配）
        │                               └── LLM + Skills
        └── Result（结果）→ Report（汇报）

Platform（平台层）
  ├── AgentTemplate 工厂（按领域/平台预置 Agent 模板）
  ├── WorkflowTemplate 工厂（按项目类型预置流程模板）
  ├── Skills 库（OpenClaw 格式兼容，可快速导入）
  └── LLM 配置中心（Provider、ModelConfig、Credential）
```
