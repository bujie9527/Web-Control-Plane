# Cursor 协作约定文档

本文档是「AI Work Control Center 控制平面」项目与 Cursor AI 协作的基础说明，适用于后续所有会话。AI 在对话中应默认遵循本约定。

---

## 一、用户角色定位

协作对象（产品/架构方）的角色为：

| 角色 | 说明 |
|------|------|
| **产品设计者** | 关注产品目标、信息架构、业务流程、页面结构与交互，而非实现细节。 |
| **架构设计者** | 关注模块边界、分层、接口契约、可扩展性与技术路线，而非逐行代码。 |
| **系统规划者** | 关注阶段目标、做与不做的边界、优先级与后续演进，而非单次任务的代码量。 |

**不假定**对方是以代码实现为主的工程师。因此：

- 输出应优先从**目标、结构、方案、影响**出发，再进入实现。
- 避免默认进入「只给代码、不讲背景」的模式；仅在对方明确要求「直接上代码」时简化说明后给出。
- 讨论技术方案时，兼顾**产品是否清晰、架构是否合理、是否便于后续协作**。

### 固定口径（每次协作均参考）

> **用户更关心结构是否正确、风格是否统一、后续是否易扩展，不追求一次写完全部业务逻辑。**

据此：优先保证骨架与规范到位，业务逻辑可分步补齐；不因「一次写全」而牺牲结构清晰与可扩展性。

## 二、Cursor 协作方式

### 2.1 沟通语言

- **默认优先使用中文**进行回复、说明与文档撰写。
- 仅在技术关键词、目录名、文件名、接口字段名等必须保留时使用英文。

### 2.2 输出顺序

- **先**：结构、思路、方案、影响范围、可选方案对比。
- **再**：具体实现步骤或代码。
- 复杂任务**先出计划**（目标、步骤、涉及模块、风险点），经确认后再拆解执行，避免直接进入细节代码。

### 2.3 与规则的配合

- 项目在 `.cursor/rules/` 下已配置核心架构、语言与协作、UI、双后台、编码边界、接口契约等规则。
- 会话中应默认遵守这些规则；若用户临时放宽某条，以用户当次说明为准。

## 三、当前阶段目标

本阶段聚焦**企业级中控后台第一阶段骨架**，不追求完整业务闭环。

### 3.1 要先做好的事

| 目标 | 说明 |
|------|------|
| **先搭后台骨架** | 双后台（平台 / 租户）可访问、菜单与路由清晰、页面可打开，业务可先占位。 |
| **先做双后台与规范架构** | 平台后台与租户后台完全分离；页面 / 服务 / schema 分层明确；模块边界清晰。 |
| **先做 mock 驱动的真结构** | 接口契约先行，用 mock 数据驱动列表、详情、表单等真实页面结构，不依赖真实后端或第三方。 |

### 3.2 当前阶段明确不做

- 不接第三方真实平台。
- 不做真实自动执行逻辑。
- 不做复杂流程拖拽。
- 不做复杂数据库实现。
- 不把复杂业务逻辑直接写进页面。

## 四、每次任务的标准流程

无论需求大小，建议按以下流程协作，以控制范围、避免返工：

| 步骤 | 内容 | 说明 |
|------|------|------|
| **1. 理解目标** | 用简短语言复述：要达成什么、为谁服务、成功标准是什么。 | 必要时向用户确认，避免理解偏差。 |
| **2. 确认影响范围** | 列出可能涉及的文件、模块、菜单、路由、接口；标明「仅读/仅新增/会修改」。 | 不扩大范围，不顺手改无关文件。 |
| **3. 输出计划** | 给出执行计划：子步骤、顺序、每步产出、依赖关系。复杂任务附简要方案对比（若适用）。 | 便于产品/架构方判断方向是否正确。 |
| **4. 等待确认后执行** | 在用户确认计划或方向后再写代码、改配置、增删文件。 | 避免做了再返工。 |
| **5. 完成后自检** | 对照目标与影响范围做自检：是否只改了该改的、是否复用已有能力、是否保持模块边界与分层。 | 可口头或列表形式简要说明。 |

用户若明确说「直接做」「不用计划」，可简化步骤 3、4，但仍建议保留「理解目标」和「完成后自检」。

## 五、UI 与模块边界要求

### 5.1 整体 UI 风格

- **企业级中控后台**：专业、克制、长期可扩展。
- **侧边栏**：深色（如深灰/深蓝），用于主导航与品牌区。
- **内容区**：浅色背景，保证表格、表单的可读性。
- 避免花哨动效与多余装饰；组件风格统一。

### 5.2 双后台分离

- **平台后台**：租户管理、平台配置、审计等平台级能力；不承载租户内业务（如某租户的项目、流程、Agent）。
- **租户后台**：项目中心、流程中心、Agent 中心、Skills 能力库、终端中心、数据分析、系统管理等租户内业务。
- 二者路由、菜单、布局完全分离，不混在同一棵菜单树下。

### 5.3 页面统一结构

页面按统一区块组织，便于维护与扩展：

1. **标题区**：页面标题，必要时副标题或面包屑。
2. **说明区**（可选）：简短说明或引导文案。
3. **操作区**：主要操作按钮（新建、导出等）。
4. **筛选/搜索区**（列表页）：筛选条件、搜索、查询/重置。
5. **内容区**：表格、卡片、列表或表单主体。
6. **抽屉/弹窗**：详情、编辑、配置等次级内容。

新增或重构页面时，应对齐上述结构；项目详情页按「工作台」设计，而非简单表单详情。

### 5.4 组件与复用

- 不随意新增与现有能力**重复**的组件；先检索是否已有可复用或可扩展的实现。
- 保持模块边界：平台侧与租户侧不混用业务组件；通用 UI 组件可共享。

## 六、与规则、Skill 的配合

- **规则**（`.cursor/rules/*.mdc`）：约束架构、语言、UI、双后台、编码边界、接口契约等；会话中默认遵守。
- **Skill**（`.cursor/skills/*`）：标准化动作（如建列表页、工作台、模块骨架、mock、分层、布局重构）；在用户提出对应需求时优先按对应 Skill 的输入/输出与原则执行。
- 本协作约定与规则、Skill 一致；若遇冲突，以用户当次指示为准。
